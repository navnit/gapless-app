# Gapless macOS 0.1.0 Release Design

## Goal

Release Gapless `0.1.0` publicly for macOS only, with separate Apple Silicon
and Intel DMGs. Every candidate and public artifact must be built on its native
GitHub-hosted macOS runner, signed, notarized, installed, smoke-tested, and
bundle-verified before it can be downloaded.

Windows and Linux remain verification targets but are not release targets for
`0.1.0`.

## Version contract

`pubspec.yaml` is the application-version source of truth. The first public
release uses:

```yaml
version: 0.1.0+1
```

The user-facing version is `0.1.0`; `1` is the monotonically increasing native
build number. A public release tag must be exactly `v0.1.0`. Candidate input
must be exactly `0.1.0`.

A small Dart validator reads `pubspec.yaml` and rejects malformed versions or a
candidate/tag version that does not equal the user-facing version. The release
workflow runs this check before fetching engines, signing, or publishing.
Flutter builds consume the committed `pubspec.yaml` value so macOS
`CFBundleShortVersionString` and `CFBundleVersion` remain aligned with the tag.

Auto-Editor keeps its independent pinned version and checksums in
`assets/engine/manifest.json`; changing the Gapless version does not implicitly
change the engine.

## Workflow architecture

The existing all-platform tagged workflow is replaced by one focused
`.github/workflows/release-macos.yml` workflow with two entry points:

1. `workflow_dispatch` accepts a required candidate version. It builds and
   validates signed/notarized DMGs, then uploads GitHub Actions artifacts. It
   never creates or updates a GitHub Release.
2. A pushed `v*` tag repeats the complete native build and validation pipeline.
   After both architectures succeed, a publication job targets the protected
   `macos-release` GitHub environment. Approval publishes a public GitHub
   Release for that tag.

The matrix contains only:

- `macos-arm64` on `macos-14`;
- `macos-x64` on `macos-15-intel`.

The shared build path performs dependency resolution, verified engine fetch,
analysis, tests, fixture generation, release build, SBOM generation, nested
code signing, hardened-runtime packaging, notarization, bundle verification,
installed-DMG smoke testing, native editor/recovery integration tests, and
checksum generation. A matrix failure prevents publication.

Artifacts use versioned names:

- `Gapless-0.1.0-macos-arm64.dmg`;
- `Gapless-0.1.0-macos-x64.dmg`;
- `SHA256SUMS`;
- per-architecture SPDX SBOMs;
- build instructions and third-party notices.

## Public access and approval

The repository is public, so an approved GitHub Release is available without a
GitHub account at:

`https://github.com/navnit/gapless/releases/latest`

Before any `v0.1.0` tag is pushed, repository administrators must create the
`macos-release` environment in GitHub Settings and configure at least one
required reviewer. The publication job declares that environment and receives
only `contents: write`; candidate/build jobs receive only `contents: read`.

The publication job downloads only artifacts produced by its own successful
tag run, consolidates their checksums, and creates the public release with
automatically generated release notes. Candidate runs cannot reach this job.

The README links to the latest public release and identifies `0.1.0` as a
macOS-only release. It continues to describe Windows and Linux as planned
targets rather than downloadable products.

## Signing and notarization

Both candidate and tagged builds use the existing production signing inputs:

- `MACOS_P12_BASE64`;
- `MACOS_P12_PASSWORD`;
- `MACOS_KEYCHAIN_PASSWORD`;
- `MACOS_NOTARY_KEY_BASE64`;
- `MACOS_NOTARY_KEY_ID`;
- `MACOS_NOTARY_ISSUER`;
- `MACOS_SIGN_IDENTITY`.

The workflow must fail with a direct missing-secret error before decoding or
invoking signing tools. Nested Auto-Editor and process-host binaries are signed
before the app. The DMG is notarized and stapled. Validation includes
`codesign --verify`, `spctl`, bundle verification, mounting the DMG, running the
installed app's deterministic smoke test, and re-verifying the mounted bundle.

No signing or notarization claim is made from local debug builds. Distribution
readiness is established only by the exported workflow artifacts and their
recorded checks.

## Failure behavior

- A version mismatch fails before artifact work begins.
- A missing engine, checksum mismatch, signing failure, notarization failure,
  bundle-verifier failure, smoke-test failure, or integration-test failure
  prevents that matrix job from uploading a releasable DMG.
- Publication requires both architecture jobs; partial releases are forbidden.
- Manual candidates never publish, even when every check passes.
- Windows and Linux jobs cannot be started by a `v0.1.0` tag.
- The release tag is immutable. A defective release is corrected with a new
  patch version rather than replacing assets under `v0.1.0`.

## Testing strategy

Automated Dart tests cover version parsing, build-number parsing, exact
candidate/tag equality, malformed inputs, and mismatch diagnostics. Workflow
contract tests assert:

- both triggers and their separation;
- the two macOS matrix entries and absence of Windows/Linux release entries;
- version validation before build/signing steps;
- candidate publication exclusion;
- protected-environment publication for tag runs only;
- versioned artifact names;
- signing, notarization, installed smoke, bundle-verifier, and integration-test
  commands;
- least-privilege job permissions.

Repository verification runs formatting, workflow contract tests, the complete
Flutter test suite, analyzer, shell syntax checks, and `git diff --check`.
GitHub PR verification must remain green on macOS, Windows, and Linux before
the workflow is merged.

## Release procedure

1. Merge the implementation PR after all PR checks pass.
2. Configure and verify the `macos-release` protected environment and macOS
   signing/notarization secrets.
3. Run the manual candidate workflow for `0.1.0` from the merged `main` commit.
4. Download and install both candidate DMGs on Apple Silicon and Intel Macs;
   complete the manual editor checklist.
5. Create annotated tag `v0.1.0` on that exact `main` commit and push it.
6. Review the tag workflow's checks and artifacts.
7. Approve the `macos-release` environment gate.
8. Confirm the public release, both DMG downloads, checksums, SBOMs, release
   notes, and `releases/latest` link.

Creating or pushing `v0.1.0` is deliberately outside the implementation PR.
It happens only after candidate validation and explicit release approval.
